# Runbook i18n — Internacionalización del Frontend

Guía operacional para gestionar las traducciones del frontend de Crezer.

## Stack de i18n

- **Librería**: `i18next` + `react-i18next` + `i18next-browser-languagedetector`
- **Idioma activo**: `es` (español) — idioma por defecto y fallback
- **Idioma secundario**: `en` (inglés)
- **Configuración**: `resources/js/i18n.ts`
- **Archivos de traducción**: `resources/js/locales/es.json` y `resources/js/locales/en.json`
- **Interpolación**: `{{variable}}` — por ejemplo `"Editar perfil de {{name}}"`

---

## Estructura de `resources/js/locales/es.json`

El archivo está organizado en **namespaces anidados** por dominio funcional. Total: ~970 líneas.

```
es.json
├── common           — Labels reutilizables: save, cancel, delete, loading, etc.
├── nav              — Items del menú lateral principal
├── nav_payroll      — Sub-menú de Nómina
├── nav_pos          — Sub-menú de Punto de Venta
├── nav_ei           — Sub-menú de Facturación Electrónica
├── user_menu        — Menú del usuario (settings, logout)
├── dashboard        — Panel principal y KPIs
├── appointments     — Citas: listado, creación, edición, estados, modales
├── services         — Servicios: CRUD completo, comisiones
├── employees        — Colaboradores: CRUD, schedule anidado
├── clients          — Clientes: CRUD, tabs de fidelización
├── schedule         — Horarios semanales de empleados
├── settings         — Ajustes del negocio
├── auth             — Login, registro, verificación
├── payroll          — Nómina: períodos, reportes, ajustes
├── pos              — Punto de Venta: tickets, turnos
├── electronic_invoice — Facturación electrónica DGII
└── errors           — Mensajes de error genéricos (404, 403, 500)
```

### Ejemplo de estructura de un namespace

```json
{
  "appointments": {
    "title": "Citas",
    "new": "Nueva cita",
    "status_pending": "Pendiente",
    "status_confirmed": "Confirmada",
    "cancel_modal_description": "¿Estás seguro de que deseas cancelar la cita de {{client}}?",
    "add_service": {
      "title": "Agregar servicio",
      "submit_btn": "Agregar servicio"
    }
  }
}
```

Los namespaces pueden anidarse un nivel (por ejemplo `appointments.add_service.title`).

## Cómo usar traducciones en componentes React

```tsx
import { useTranslation } from 'react-i18next';

export default function AppointmentCard() {
    const { t } = useTranslation();

    return (
        <div>
            <h1>{t('appointments.title')}</h1>
            <p>{t('appointments.cancel_modal_description', { client: 'Ana López' })}</p>
        </div>
    );
}
```

La función `t('namespace.key')` resuelve la clave desde el idioma activo. Si la clave no existe, devuelve la clave misma como fallback visible.

## Cómo agregar una nueva traducción

1. **Ubicar el namespace correcto** en `resources/js/locales/es.json`. Si la clave pertenece a un módulo nuevo, crear un namespace nuevo al final del archivo.

2. **Agregar la clave en `es.json`**:

```json
{
  "appointments": {
    "new_key": "Texto en español"
  }
}
```

3. **Agregar la clave en `en.json`** con la traducción en inglés:

```json
{
  "appointments": {
    "new_key": "English text"
  }
}
```

   Si `en.json` no tiene la clave, `i18next` usa `es.json` como fallback (configurado en `i18n.ts`).

4. **Usar la clave en el componente**:

```tsx
t('appointments.new_key')
```

5. **Construir el frontend**:

```bash
ddev exec --dir /var/www/html/backend npm run build
```

## Cómo cambiar un string visible en la UI

Para modificar un texto que ya aparece en pantalla:

1. **Identificar la clave**: buscar el texto en `es.json`:

```bash
grep -n "Guardar cambios" resources/js/locales/es.json
# Output: 4:    "save_changes": "Guardar cambios",
```

2. **Editar el valor** en `resources/js/locales/es.json`:

```json
{
  "common": {
    "save_changes": "Actualizar"
  }
}
```

3. **Reconstruir el frontend**:

```bash
ddev exec --dir /var/www/html/backend npm run build
```

El cambio aplica globalmente en todos los componentes que usen `t('common.save_changes')`.

---

## Reconstruir el frontend después de editar traducciones

Cualquier cambio en los archivos `.json` de locales requiere un rebuild de assets:

```bash
# Build de producción (para deploy o staging)
ddev exec --dir /var/www/html/backend npm run build

# Modo desarrollo (hot reload automático al guardar)
ddev exec --dir /var/www/html/backend npm run dev

# Build completo incluyendo PHP (recomendado en dev)
ddev exec --dir /var/www/html/backend composer run dev
```

En producción, después de editar y hacer deploy:

```bash
npm ci
npm run build
php artisan view:cache
```

## Detección de idioma

`i18next-browser-languagedetector` detecta el idioma del usuario en este orden:

1. `localStorage` — si el usuario cambió el idioma manualmente
2. `navigator.language` — idioma del navegador

El idioma activo por defecto y fallback es `es`. No hay selector de idioma en la UI actualmente; el sistema sirve en español para todos los usuarios del negocio.

## Interpolación de variables

Las claves pueden recibir variables dinámicas con la sintaxis `{{variable}}`:

```json
{
  "appointments": {
    "cancel_modal_description": "¿Estás seguro de que deseas cancelar la cita de {{client}}?",
    "edit_subtitle": "Actualizar datos de la cita de {{client}}"
  }
}
```

Uso en componente:

```tsx
t('appointments.cancel_modal_description', { client: appointment.client.name })
```

Las variables deben coincidir exactamente (case-sensitive) entre la key del JSON y el objeto de interpolación.

## Agregar un idioma nuevo (futuro)

Si se requiere soporte para un tercer idioma (por ejemplo `pt` para portugués):

1. Crear `resources/js/locales/pt.json` copiando la estructura de `es.json`.
2. En `resources/js/i18n.ts`, agregar:

```ts
import pt from './locales/pt.json';

i18n.use(...).init({
    resources: {
        es: { translation: es },
        en: { translation: en },
        pt: { translation: pt },  // <-- agregar
    },
    ...
});
```

3. Reconstruir con `npm run build`.
